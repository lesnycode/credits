# Credits: добавить трек на сайт работ

Добавляет одиночный трек на **credits.podlesnytwins.com** (портфолио сведения/мастеринга Podlesny Twins) **по одной Spotify-ссылке**: скрипт сам тянет из Spotify артиста, название, год и обложку, сам подбирает жанр по уже имеющимся плиткам артиста, вставляет плитку, пишет роль и перегенерирует SEO-страницы + sitemap. Память: `project_works_portfolio`, `project_sound_design_blog`.

Локально: `~/works-site` (репо `lesnycode/credits`). Один статический `index.html`, весь HTML/CSS/JS инлайн. SEO-страницы `track/<slug>/` и хаб `track/` **генерируются** скриптом `build_who_mixed.py` — руками их не трогать.

## Что автоматически, что нет
Из Spotify скрипт достаёт сам (переопределять только если ошибся):
- **Артист** — из embed-метаданных (для фита все артисты через «, »).
- **Название** и **обложку** (хеш) — из oEmbed.
- **Год** — из `releaseDate` embed-страницы.
- **Жанр** — берёт `data-g` существующих плиток того же артиста (единообразие корзины). Если артист **новый** и жанра нет — скрипт печатает `GENRE_UNKNOWN` и **ничего не пишет**; тогда **агент сам определяет жанр** по своим знаниям об артисте (4 корзины: рэп→`HIPHOP`, поп/эстрада→`POP`, авторская/инди→`INDIE`, рок→`ROCK`) и перезапускает с `--genre`. Пользователя дёргать не нужно.
- **Иноагент** — авто по встроенному набору `IA` (сейчас Morgenshtern); ставит `*`.

Скрипт НЕ угадывает (безопасные дефолты, менять по словам пользователя):
- **Роль**: свели → `mix` (дефолт), свели и мастерили → `mm`, только мастеринг → `master`. Влияет на текст SEO: «Кто свёл» / «Кто мастерил».
- **Награда** `--award GRAM|TOP1` — фильтры «Золотой граммофон» / TOP-1.

## Процедура

Один скрипт, пять подкоманд. Путь к нему — `scripts/add_track.py` в **базовой директории этого скила** (её абсолютный путь Claude Code печатает в шапке при вызове — «Base directory for this skill: …»); задать `SCRIPT="<эта база>/scripts/add_track.py"`. Скрипт работает с репо `~/works-site` (флаг `--repo PATH`, если репо в другом месте). **CSV/таблицу с плейлиста выгружать не нужно** — хватает Spotify-ссылки (трек / альбом / плейлист).

### 1. Добавить трек(и) — `add`
Обычно только ссылка (или несколько за раз — билд прогонится **один раз** в конце):
```bash
python3 "$SCRIPT" add "<url1>" "<url2>" ... --json
```
Скрипт на каждую: тянет метаданные из Spotify → хеш обложки → жанр по артисту → плитка в **точном** формате сайта (srcset 300/640, alt `Артист — Трек · сведение и мастеринг Podlesny Twins`) → рядом с плитками того же артиста → роль в `roles.json`. В конце `build_who_mixed.py` один раз. `--json` печатает `RESULT {...}` на каждый трек — свериться с артистом/годом/жанром.

- Подкоманду `add` можно **опустить**: `python3 "$SCRIPT" "<url>"` = `add`.
- **Жанр на конкретную ссылку в пачке:** суффикс `@GENRE` — `"...track/ID1@POP" "...track/ID2@HIPHOP"`.
- Флаги (на весь вызов): `--genre POP` (только для нового артиста), `--role mm`, `--award TOP1`, `--ia`/`--no-ia`. Переопределения `--artist/--title/--year` — **только с одной ссылкой**.
- **`--dry-run`** — показать `RESULT` и ничего не писать (проверить авто-метаданные до билда). Рекомендуется первый прогон новых артистов.
- Дедуп: трек с этим `data-id` уже на сайте → пропуск.
- **`GENRE_UNKNOWN` (exit 3):** новый артист, скрипт **ничего не записал** (атомарно, даже в пачке). Определить жанр по знанию артиста → перезапустить с `--genre` или суффиксом `@GENRE`. Пользователя не дёргать.

### 2. Добавить альбом — `album`
```bash
python3 "$SCRIPT" album "<album-url>" [--genre POP] --json
```
Тянет треклист из Spotify → добавляет объект в JS-массив `ALBUMS` (`{artist,name,cover,tracks:[{id,title,year}]}`) + плитку `.tile.album` c `openAlbum(N)` (N = новый индекс, существующие не сдвигаются) и бейджем «N треков» → роль для **каждого** трека в `roles.json` → билд. Год берётся из релиза (если embed альбома не отдал — из первого трека; можно задать `--year`). Артист/название — проверить по `RESULT`, при латинице вместо кириллицы задать `--artist`.

### 3. Синк плейлиста — `sync` (только ссылка на плейлист)
```bash
python3 "$SCRIPT" sync "<playlist-url>" --json
```
Тянет треклист плейлиста из Spotify → **добавляет только новые** треки (дедуп по тому, что уже на сайте — и как плитка, и как трек внутри альбома). Каждый новый: метаданные + обложка + жанр по артисту, как в `add`, билд один раз в конце.

- **`sync` не атомарен** (в отличие от `add`): всё, что распозналось (артист уже есть на сайте → жанр авто, либо задан `--genre`), пишется; треки **новых** артистов без жанра выводятся списком-подсказкой `add "…track/ID@<GENRE>" # Артист — Трек` и билд идёт по остальным. Exit 3 сигналит, что остались треки без жанра.
- Дальше по этому списку: определить жанр каждого нового артиста по знанию → добить командой `add "<url>@GENRE" ...` (можно пачкой). На следующих синках этот артист уже известен → подтянется авто.
- **`--dry-run`** — показать, сколько новых/распознано/без жанра, ничего не писать. Хорошо гонять первым.
- **Лимит embed ~50 треков.** Без Spotify API страница плейлиста отдаёт первые ~50. Для большого рабочего плейлиста: держать новое ближе к верху, либо вести маленький «стейджинг»-плейлист под добавление. Уже добавленные всё равно пропускаются — важны только новые.
- Роль на весь синк — `--role` (дефолт `mix`); разные роли по трекам не угадать, править потом точечно.

### 4. Убрать трек — `remove`
```bash
python3 "$SCRIPT" remove "<url-или-id>" [...] 
```
Снимает плитку(и) одиночных треков + записи из `roles.json` → билд (он же удаляет осиротевшие `track/<slug>/`). Треки из альбома отдельной плитки не имеют — их не удалить так (правится объект в `ALBUMS` вручную).

### 5. Только перегенерировать — `build`
```bash
python3 "$SCRIPT" build
```
Голый прогон `build_who_mixed.py`: пересобирает SEO-страницы/хаб/sitemap и **чистит осиротевшие** `track/*` (например, оставшиеся после ручного удаления плитки — как страница Воробьёвой из TODO).

Формат одиночной плитки (для справки/ручной правки):
```html
<button class="tile" data-g="POP" data-id="ID" onclick="play(this)"><img src="https://i.scdn.co/image/ab67616d0000b273HASH" alt="ARTIST — TITLE · сведение и мастеринг Podlesny Twins" loading="lazy" srcset="https://i.scdn.co/image/ab67616d00001e02HASH 300w, https://i.scdn.co/image/ab67616d0000b273HASH 640w" sizes="(max-width:560px) 46vw, 200px" decoding="async"><span class="ov"><i class="pl">&#9654;</i></span><span class="meta"><span class="art">ARTIST</span><span class="trk">TITLE</span><span class="yr">YEAR</span></span></button>
```
Обложка: хеш из oEmbed `thumbnail_url` (`ab67616d00001e02<hash>`), на сайте — `ab67616d0000b273<hash>` (640) + `ab67616d00001e02<hash>` (300).

## После записи

### Проверить порядок плитки
**Правило порядка (важно): новые треки — НИКОГДА не наверх.** Верхние ~20–30 плиток пользователь отобрал вручную как самые сильные миксы — их порядок не трогать. Новый артист садится в **середину** грида (`insert_tile` в `add_track.py` вставляет после срединной плитки, не после открытия грида). Существующий артист — рядом со своими плитками (группировка). Если авто-позиция всё же оказалась выше топ-30 или трек должен встать в конкретное место — сдвинуть `<button>` вручную вниз, в середину.

Наверху грида закреплён порядок (память `project_works_portfolio`): PIZZA → Билан → MACAN&NAVAI → T-Fest → Андрей Катиков. Если новый трек должен попасть в закреплённую группу — поправить позицию `<button>` в гриде вручную. Счётчики фильтров (`.fc`) пересчитываются в JS сами — трогать не надо.

### Что перегенерировал build_who_mixed.py
- `track/<slug>/index.html` — SEO-страница трека (title/description/canonical/OG, JSON-LD `MusicRecording`+`FAQPage`+`BreadcrumbList`, Spotify-эмбед). Slug — транслит `артист-трек`.
- `track/index.html` — хаб «Кто свёл», сгруппирован по артисту.
- `sitemap.xml` — все URL с **честным пер-slug `lastmod`** (см. `seo_state.json` ниже): дата меняется только у трека, у которого реально изменился контент, у остальных сохраняется прежняя. Не «всё сегодня».
- Футер `index.html` — ссылка «Полный список треков» (идемпотентно).

Роль определяет текст: `mix`/`mm` → «Кто свёл», `master` → «Кто мастерил». Опц. история трека — в `notes.json` по slug (массив абзацев), подхватится при следующем билде.

### SEO-леджер `seo_state.json` (не сбивать SEO при полной регенерации)
Билд каждый раз **сносит весь `track/` и рендерит все страницы заново** (детерминированно). Чтобы это не роняло SEO, скрипт ведёт `~/works-site/seo_state.json` — коммитить в git, **руками не трогать**. Три поля:
- `ids` — `data-id → текущий slug`. Если у трека сменился slug (переименовали артиста/название задним числом), скрипт видит по `data-id`, что это тот же трек.
- `redirects` — `старый-slug → новый-slug`. На каждый ретайрнутый slug билд ставит **редирект-заглушку** `track/<старый>/index.html` (canonical + `meta refresh` + JS `location.replace` на новый URL). GitHub Pages статик — иначе старый URL отдавал бы 404 без 301, и накопленный вес пропадал. Цепочки `a→b→c` схлопываются в `a→c`; заглушки не попадают в sitemap. Осиротевший трек (совсем удалён) редиректа не получает — честный 404.
- `dates` — `slug → {hash, lastmod}`. Хэш = отпечаток контент-полей (артист/трек/год/роль/фит/альбом/обложка/notes), не всего HTML — правки CSS/шаблона `lastmod` не сбрасывают.

**Итог:** добавил трек, порядок/тексты не менял → все прочие даты и URL стабильны. Переименование трека → старый URL живёт редиректом. Файл потерялся/не закоммичен → на следующем билде леджер отстроится заново от текущего состояния (даты станут «сегодня», прошлые редиректы забудутся) — поэтому **`seo_state.json` обязательно в коммит**.

## Подводные камни
- **Сверять авто-метаданные.** Артист/год/название берутся из Spotify — глянуть строку `RESULT {...}`. Год = год релиза конкретного трека/сингла (у переизданий может отличаться). Артист приходит как на Spotify (напр. латиницей) — если на сайте у него плитки кириллицей, задать `--artist` в том же написании (иначе не сгруппируется). Для фита все артисты идут в строку через «, » (в SEO-фит-строку — все, кроме первого).
- **Жанр нового артиста** определяет агент, не пользователь: при `GENRE_UNKNOWN` выбрать корзину по знанию артиста и перезапустить с `--genre`. У артиста с разными релизами скрипт возьмёт корзину первой найденной плитки — если жанр другой, переопределить `--genre`.
- **Иноагенты** — авто по набору `IA` в скрипте (сейчас Morgenshtern); сноску в хабе/футере build-скрипт добавляет сам (его набор `IA` в `build_who_mixed.py`). Новый иноагент → дописать его в оба набора `IA` (в `add_track.py` и `build_who_mixed.py`). Принудительно: `--ia` / `--no-ia`.
- **Сеть.** Скрипту нужен доступ к `open.spotify.com` (oEmbed + embed-страница). Если embed-метаданные недоступны — год/артист не подтянутся, задать вручную флагами.
- **Альбом.** `album` добавляет объект в конец `ALBUMS` и плитку с `openAlbum(N)`, где N = новый индекс (существующие альбомы не сдвигаются — безопасно). Треклист/год берутся из Spotify; сверить `RESULT`. Feats на треках альбома — из `feats.json` (по id), при нужде дописать вручную. Удаление трека **из** альбома — правка объекта `ALBUMS` руками, `remove` их не берёт (у них нет отдельной плитки). **После правки треклиста альбома вручную поправить и `<span class="abadge">N треков</span>` на плитке этого альбома** — счётчик статичный текст в HTML, JS его не пересчитывает (в отличие от `.fc` у жанровых фильтров). Заодно снять роли/заметки удалённых треков из `roles.json`/`notes.json`, потом билд — осиротевшие `track/<slug>/` подчистятся сами.
- **Пачка (`add`) атомарна:** если хоть один трек упал в `GENRE_UNKNOWN` — не пишется НИЧЕГО, билд не запускается. Дать жанр (`@GENRE`/`--genre`) и перезапустить всю команду.
- `data-aw` управляет только фильтром наград — визуальных бейджей на карточках нет.
- Не редактировать `track/*` руками — перезапишется билдом. Менять данные в `index.html`/`roles.json`/`notes.json`, потом билд (или подкоманда `build`).
